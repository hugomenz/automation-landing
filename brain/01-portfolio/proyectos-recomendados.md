# Proyectos recomendados

## 1. Angular Design System Rollout

| Campo | Contenido |
|---|---|
| Problema de negocio | UI inconsistente, bugs repetidos y lentitud al desarrollar features |
| Stack recomendado | Angular, Storybook, Nx, TypeScript, design tokens, SCSS/Tailwind, Playwright |
| Resultado demostrable | Component library, documentación, ejemplos, tests visuales o E2E básicos |
| Cliente que atrae | SaaS, producto B2B, empresas industriales con portales Angular |
| Dificultad | Media-alta |
| Tiempo versión presentable | 5–7 días |
| Cómo explicarlo senior | “No construí componentes sueltos; diseñé un sistema para reducir coste de cambio, duplicación y discusiones repetidas entre diseño y desarrollo.” |

## 2. Internal Operations Console

| Campo | Contenido |
|---|---|
| Problema de negocio | Operaciones dependen de Excel, emails y copiar/pegar entre sistemas |
| Stack recomendado | Angular, NestJS/Node, PostgreSQL, REST API, n8n, auth básica |
| Resultado demostrable | Dashboard operativo, flujo automatizado, logs y trazabilidad |
| Cliente que atrae | Pymes, logística, ventas, soporte, operaciones internas |
| Dificultad | Media |
| Tiempo versión presentable | 4–6 días |
| Cómo explicarlo senior | “Tomé un proceso manual y lo convertí en una herramienta operativa con trazabilidad, menos errores y mejor UX interna.” |

## 3. GitHub Copilot Adoption Kit

| Campo | Contenido |
|---|---|
| Problema de negocio | Copilot comprado pero usado sin standards, prompts, governance ni medición |
| Stack recomendado | GitHub, Copilot instructions, Markdown, PR templates, examples repo |
| Resultado demostrable | Playbook, ejercicios, checklist de calidad, prompts por flujo de trabajo |
| Cliente que atrae | Equipos dev de 10–80 personas |
| Dificultad | Baja-media |
| Tiempo versión presentable | 2–4 días |
| Cómo explicarlo senior | “No vendo prompts; vendo adopción controlada de AI-assisted development con guardrails, revisión crítica y workflows reproducibles.” |

## 4. Customer Support Triage Automation

| Campo | Contenido |
|---|---|
| Problema de negocio | Tickets repetitivos, clasificación manual y tiempos altos de respuesta |
| Stack recomendado | n8n/Make, API de helpdesk, Slack/Email, LLM API, base de conocimiento simple |
| Resultado demostrable | Workflow que clasifica, resume, enruta y genera borradores con humano en el loop |
| Cliente que atrae | SaaS, e-commerce B2B, soporte técnico, agencias |
| Dificultad | Media |
| Tiempo versión presentable | 3–5 días |
| Cómo explicarlo senior | “Diseñé un workflow con control humano, fallback, logging y control de coste; no un chatbot de demo.” |

## 5. Public Data Monitor / Legal Scraping Workflow

| Campo | Contenido |
|---|---|
| Problema de negocio | Revisión manual de licitaciones, cambios públicos, listados o fuentes abiertas |
| Stack recomendado | Python, Playwright, APIs públicas, n8n, base de datos simple, email digest |
| Resultado demostrable | Pipeline que extrae, normaliza, deduplica y notifica cambios relevantes |
| Cliente que atrae | Consultoras, procurement, legal ops, real estate, energía, research |
| Dificultad | Media-alta |
| Tiempo versión presentable | 4–7 días |
| Cómo explicarlo senior | “No hice scraping por scraping; monté un workflow legal, trazable y orientado a decisiones.” |
